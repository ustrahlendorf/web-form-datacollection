# Environment Configuration Guide

This document explains how to configure environment variables and secrets for the Data Collection Web Application.

## Overview

The application requires configuration of several AWS resources and endpoints:

1. **Lambda Environment Variables**: DynamoDB table name
2. **Frontend Configuration**: API endpoint, Cognito credentials, CloudFront domain
3. **CORS Policy**: CloudFront domain for API access
4. **Cognito Callbacks**: CloudFront domain for OAuth redirects

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│                    Configuration Flow                        │
└─────────────────────────────────────────────────────────────┘

1. Deploy CDK Infrastructure
   ↓
2. Retrieve CloudFront Domain from Stack Outputs
   ↓
3. Update Cognito Callback URLs with CloudFront Domain
   ↓
4. Update API Gateway CORS Policy with CloudFront Domain
   ↓
5. Generate Frontend .env File with All Configuration
   ↓
6. Build Frontend with Configuration
   ↓
7. Deploy Frontend to S3/CloudFront
```

## Step-by-Step Configuration

### Step 1: Deploy CDK Infrastructure

Deploy all CDK stacks to AWS:

```bash
cd infrastructure/
cdk deploy --all --require-approval never
```

This creates:
- Cognito User Pool and Client
- DynamoDB table
- API Gateway HTTP API
- Lambda functions
- S3 bucket and CloudFront distribution

**Outputs from this step:**
- CloudFront Domain Name
- API Endpoint URL
- Cognito User Pool ID
- Cognito Client ID
- Cognito Hosted UI Domain

### Step 2: Configure Lambda Environment Variables

Lambda functions automatically receive the DynamoDB table name via environment variables set in the CDK stack:

```python
# In api_stack.py
environment={
    "SUBMISSIONS_TABLE": table_name,
}
```

**Lambda Functions Configured:**
- `submit_handler`: Receives `SUBMISSIONS_TABLE` environment variable
- `history_handler`: Receives `SUBMISSIONS_TABLE` environment variable
- `recent_handler`: Receives `SUBMISSIONS_TABLE` environment variable

### Step 3: Configure Frontend Environment Variables

The frontend requires several configuration values. These are managed through a `.env` file:

```bash
# API Gateway endpoint URL
API_ENDPOINT=https://xxxxx.execute-api.eu-central-1.amazonaws.com

# Cognito configuration
COGNITO_DOMAIN=https://data-collection-dev.auth.eu-central-1.amazoncognito.com
COGNITO_CLIENT_ID=xxxxxxxxxxxxxxxxxxxxx
COGNITO_USER_POOL_ID=eu-central-1_xxxxxxxxxxxxx
COGNITO_REDIRECT_URI=https://d111111abcdef8.cloudfront.net

# CloudFront domain for CORS
CLOUDFRONT_DOMAIN=d111111abcdef8.cloudfront.net

# Environment name
ENVIRONMENT=dev
```

### Step 4: Configure CORS Policy

The API Gateway CORS policy must include the CloudFront domain:

```python
# In api_stack.py
cors_origins = [
    "http://localhost:3000",      # Local development
    "http://localhost:8000",      # Local development (alternative)
    "https://d111111abcdef8.cloudfront.net",  # Production CloudFront
]
```

### Step 5: Configure Cognito Callbacks

Cognito OAuth callbacks must include the CloudFront domain:

```python
# In cognito_stack.py
callback_urls = [
    "http://localhost:3000",
    "http://localhost:8000",
    "https://d111111abcdef8.cloudfront.net",
]

logout_urls = [
    "http://localhost:3000",
    "http://localhost:8000",
    "https://d111111abcdef8.cloudfront.net",
]
```

## Automated Configuration

### Using setup-env.sh

The `setup-env.sh` script automates the retrieval of configuration from CDK stack outputs:

```bash
cd frontend/
bash setup-env.sh dev
```

This script:
1. Retrieves CloudFront domain from `DataCollectionFrontend-dev` stack
2. Retrieves API endpoint from `DataCollectionAPI-dev` stack
3. Retrieves Cognito configuration from `DataCollectionCognito-dev` stack
4. Generates `.env` file with all values
5. Displays configuration summary

### Using deploy-with-config.sh

The `deploy-with-config.sh` script automates the entire deployment process:

```bash
cd infrastructure/
bash deploy-with-config.sh dev
```

This script:
1. Deploys all CDK stacks
2. Retrieves CloudFront domain
3. Updates Cognito callback URLs with CloudFront domain
4. Displays next steps for frontend deployment

## Troubleshooting

### Configuration Not Loading

**Problem**: Frontend shows "Failed to load recent submissions" or API errors

**Solution**:
1. Verify `.env` file exists: `cat frontend/.env`
2. Verify `config.js` was generated: `cat frontend/build/config.js`
3. Check browser console for configuration errors
4. Verify API endpoint is correct: `echo $API_ENDPOINT`

### CORS Errors

**Problem**: Browser console shows CORS errors

**Solution**:
1. Verify CloudFront domain is in CORS policy
2. Redeploy API stack with updated CORS origins
3. Check that frontend is served from CloudFront domain

### Cognito Login Fails

**Problem**: Cognito login redirects to error page

**Solution**:
1. Verify callback URLs in Cognito User Pool Client
2. Verify CloudFront domain is in callback URLs
3. Check that redirect URI matches CloudFront domain
4. Verify Cognito domain is correct

### Environment Variables Not Set

**Problem**: Lambda functions fail with "SUBMISSIONS_TABLE not found"

**Solution**:
1. Verify Lambda environment variables are set in CDK stack
2. Check CloudWatch Logs for Lambda execution errors
3. Redeploy API stack to ensure environment variables are set

## Security Considerations

### CORS Policy

- ✅ Only allows specific origins (localhost for dev, CloudFront for prod)
- ✅ Restricts to necessary HTTP methods (GET, POST, OPTIONS)
- ✅ Restricts to necessary headers (Content-Type, Authorization)

### Cognito Callbacks

- ✅ Only allows specific callback URLs
- ✅ Uses HTTPS for production URLs
- ✅ Prevents open redirect vulnerabilities

### Environment Variables

- ✅ Sensitive values (API endpoints, IDs) are environment-specific
- ✅ No hardcoded secrets in code
- ✅ Configuration is generated at build time

### Lambda Environment Variables

- ✅ DynamoDB table name is set via CDK
- ✅ Lambda execution role has least-privilege permissions
- ✅ No hardcoded credentials in Lambda code
